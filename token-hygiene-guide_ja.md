# LLM トークン衛生（文脈エコノミー）

リポ非依存の汎用ノウハウ。能力の低い LLM ほど効く（自分で文脈を節約できないため）。
正本はここ。各リポの CLAUDE.md には短い実ルールだけを inline し、理由はここを参照する。

## 原則：情報を失わずトークンだけ削る

LLM はステートレス。毎ターン会話全体（自分のプロンプト・LLM の応答・全ツール結果）が
入力として再送・再処理され、課金される。「メッセージ token 数」＝今打った分ではなく、
セッションで積んだ全部の累計を毎回払い直す。よって基本戦略は「捨てる」ではなく
「ディスクに全部残し、文脈には必要分だけ載せる」。読みたければ読めるが既定では食わない状態を作る。

## ビルド/テストの定型

```
npm run build > build.log 2>&1 || tail -n 80 build.log
```

- 成功（exit 0）：何も出力されない＝文脈ゼロ。全ログは build.log に丸ごと残る。
- 失敗（非0）：末尾80行だけ文脈に出る。足りなければ grep -E "error|warn" build.log 等で後から掘る。
- 注意：tail は 0 を返すので、この一行の終了コードは成功扱いになる。失敗を後段（&& チェーン）へ
  伝えたい場合はここで握り潰さないよう別途 exit code を見ること。

## 補助

- 冗長処理（ビルド・テスト・ログ解析）は subagent に委譲し、本体には要約だけ返す。
- run→確認を1ターンに束ね往復を減らす（往復ごとに全履歴を払い直すため）。成功報告は短く。
- 用件が変わったら resume せず新規 or /clear。無関係な作業に古い文脈を引きずらない。
- 長くなったら /compact、ただしタイミングを選ぶ。トリガは「冗長文脈が役目を終え、かつ次は
  別サブタスクへ移る」合流点（ビルド成功は高確率で一致＝目印になる）。順序は
  NOTES 書き戻し → 続行するなら compact、店じまいなら何もしない（次回 fresh）。
  compact 自体が会話全文を読む有料の要約呼び出しで温キャッシュも切れるため、続行時だけ元が取れる。
  反応的（忘れ始めてから）でなく健全な状態で畳むほど要約の質が高い。

## 計測

/context（今積まれている内訳）と /cost・/usage（消費）で原因を特定してから手を打つ。
