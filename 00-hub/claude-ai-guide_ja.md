# claude.ai プロジェクト運用ガイド

claude.ai 側のプロジェクト（マイクラリモコンのナレッジ）を、このリポと共存させて運用するための手引き。**正本はこのリポ、claude.ai はその下流（スナップショット）。**

## 0. プロジェクトは1つに（規模が変わるまで）

claude.ai プロジェクトは**スコープで分けず、リポ全体を1プロジェクトで運用する**。

- 理由: このプロジェクトの価値は横断（protocol が plugin/python/scratch にまたがる、結節点 `debug-session-design` 等）。スコープ分割は AI が一度に見られる文脈を1スライスに狭め、**いちばん効かせたい横断文脈を切り落とす**。
- claude.ai プロジェクトはフラットで相互参照不可（`knowledge-repo-design_ja.md` §4）。分けると `DECISIONS` の横断ログを片方から見られず、ハブ&スポークが崩れる。
- 容量上限に達していない間は、手動同期の差し替え先を増やさない意味でも1つが有利。
- 分けても真実の源は割れない（正本はリポ単一）。分割コストは情報の分断ではなく**文脈の分断**で、それがこの横断重視プロジェクトでは一番痛い。

### 将来、分割を考える合図

- ナレッジが容量上限に当たったとき。切れ目は**番号スキームに従う**：`00-hub`＋`10番台`（protocol/plugin/python/scratch/evidence、相互参照が濃い）は一緒に残し、比較的独立した `20-教材` / `30-広告宣伝` から切り出す。
- あるスコープが別の協力者・別読者を持つ独立ワークストリームになり、開発コアとほぼ交わらなくなったとき。
- 分割する場合は、各プロジェクトの instructions に「**横断決定の正本は単一リポの `00-hub/DECISIONS_ja.md`**」を必ず明記する。

## 1. 現状の同期方式（手動）

MCP の GitHub コネクタは**未設定**。当面は手動の往復で運用する。

- **リポ → claude.ai**：リポを変更・コミットしたら、claude.ai のプロジェクトナレッジの該当ファイルを手動でアップロード差し替え。
- **claude.ai → リポ**：claude.ai の会話で決定や新文書が生まれたら、手でダウンロード（コピー）してリポに反映 → コミット。
- claude.ai の回答は常に「**最後にアップロードした時点のスナップショット基準**」と意識する。

## 2. instructions と knowledge の振り分け

claude.ai プロジェクトには「常時効く instructions」と「参照される knowledge」がある。役割で分ける。

| 区分 | 置き場 | 中身 |
| --- | --- | --- |
| 運用ルール（常時効かせる） | Project の instructions 欄 | `00-hub/claude-ai-instructions_ja.md` の中身を貼る |
| 設計記録（参照する） | project knowledge | 各スポークの `_ja.md` 設計記録（README を除く）＋ `00-hub/*` ＋ 全体像参照用に `CLAUDE.md` |
| 実装スナップショット | 原則載せない | `12-python-client/reference/` のコードは嵩むだけ。実装議論時のみ選択的に |

- **`CLAUDE.md` は instructions 欄には貼らない**（`@import`・git 等の Code 固有指示が claude.ai を誤誘導する）。instructions は翻案版 `claude-ai-instructions_ja.md` を使う。ただし**全体像把握の参照として knowledge には入れてよい**（Code 固有指示は claude.ai には当てはまらない、と理解する）。
- **README は claude.ai に上げない**（リポ/IDE のフォルダ道案内用。フラット名前空間では `README_ja.md` が全スポークで名前衝突する）。**例外として `00-hub/README_ja.md`（ディレクトリ索引）だけ上げる**。
- ディレクトリの番号接頭辞をそのまま反映すれば、リポと 1:1 で見通せる（ファイル名はトピック-ジャンル `<topic>-<genre>_ja.md`）。

## 3. 更新ループ

```text
①議論・決定（どの surface でもよい）
   ↓
②リポに書く（該当スポーク＋横断なら 00-hub/DECISIONS_ja.md に1行）
   ↓
③repository-owning surface でコミット／push
   ↓
④claude.ai のナレッジを差し替え（手動アップロード）
```

最大の落とし穴は ①②間の「**転記ギャップ**」。snapshot-only surface で出た結論を書き戻し忘れると消える。検証可能な repository-owning surface へ搬送し、②③まで閉じる。

## 4. 共存・競合のベストプラクティス

- **書き込み口を一本化**：同時に複数 surface から repository を書かない。claude.ai Project は現状 snapshot-only とし、repository-owning surface へ搬送する。
- **IDE 間（VS Code / IntelliJ）は同じローカルリポ＝同じ作業ツリー**。2つで同時に書かせない、切り替え前にコミット、複数 PC は push/pull 規律（GitHub が待ち合わせ点）。
- **自動メモリの分裂に注意**：claude.ai の自動生成メモリと repository の正典は別物（`00-hub/knowledge-repo-design_ja.md` §4）。効かせたいことは必ずファイルに。

## 5. 役割分担

| | repository-owning surface | claude.ai Project（snapshot-only） |
| --- | --- | --- |
| 向く作業 | 決定の記録、構成変更、リポに残る文書の起草、コミット | 設計の壁打ち、全ナレッジ横断の検討、記事ドラフト、外出先/モバイル |
| 強み | 決定→記録→コミットをその場で閉じる | 知識が常時文脈に入る／ローカル環境不要 |

最優先価値が「判断理由を残す」である以上、検証可能で repository に着地できる surface は転記ギャップが少ない。

## 6. 将来：MCP GitHub コネクタ導入時に変わること

MCP の GitHub コネクタ＋トークンを設定すると、claude.ai から本リポを直接 read/write できる（手動往復を自動化）。導入時の方針：

- **トークンは絞る**：Fine-grained PAT、対象は本リポのみ、権限は Contents: Read/write のみ、有効期限は短め（`00-hub/llm-agent-boundary-guide_ja.md` §3 準拠）。実値はファイルに書かない。
- 直接 write が可能になっても、**単一書き込み口の規律は維持**（同時書き込みを避ける）。claude.ai が DECISIONS に直接1行追記できるようになれば転記ギャップは消える。
- 導入したら本ガイド §1、knowledge-repo-design §4/§6、DECISIONS を更新する。

## 7. セッション・ループ（往来は自由・着地は固定）

どの surface でも、判断履歴を取りこぼさず回すための定型。当初の線形（壁打ち→確定→渡す→実行）を、実態（入口どこでも・渡り歩き・実行も検討/試行）に合わせて **sink/gate モデル** に組み替えた版（`2026-06-23-01`）。語彙は `2026-07-16-07` で「**確約 / 捕捉 / 工房**」に統一（旧称 Sink 1 / Sink 2・「確約面 / 捕捉面」の「面」は廃止。サーフェスは「作業する場所」専用）。

- **経路は自由**：入口はどのサーフェス・どのハブ/スポークでもよく、ハブ↔スポークを渡り歩き、実行フェーズで設計を考え直してもよい（ループバック前提）。各所・各工程で新情報・前提崩れ・再優先が起きる。
- **縛るのは経路でなく着地点**：出てきた判断・芽が必ず既知の置き場（sink）に落ちること、ここだけ規制する。リポの核心価値（判断理由を残す・SSOT）を守るのは道順の固定でなく**着地の固定**。線形を強制しても守られず形骸化し、他の規律の信用まで毀損する。

用語は [terms-glossary_ja.md](terms-glossary_ja.md) を正典とする（sink＝着地場所／gate＝着地条件／park＝捕捉への退避／確約・捕捉・工房＝裁き状態 等）。

### 現在地を問い合わせる

人間はいつでも「セッションループの現在地は？」「今どの step/status？」と聞ける。agent は長い要約ではなく、次の短い形で返す。
step/status の正典語と machine key は [terms-glossary_ja.md](terms-glossary_ja.md) §セッションループ状態名に従う。

```markdown
## セッションループ現在地

- step: <正典語> (`<machine_key>`)
- status: <正典語> (`<machine_key>`)
- sink: 確約 / 捕捉 / なし / 未判定
- 根拠: <読んだ SSOT・検証結果・搬送票などを1行>
- 次の一手: <今やるべき最小の行動を1行>
```

返答時の原則：

- step は「今どの工程として扱うか」の投影であり、実際の経路を固定しない。実装中に検討へ戻る、着地確認から差分メモへ戻る、などのループバックは正常。
- status は primary を1つだけ出す。複数に見える場合は、次の一手を決めるうえで最も支配的な阻害条件を選ぶ。
- `着地待ち` は knowledge リポ側の ID 採番・配置・commit/push 待ち、`確認待ち` は着地後に元リポ側または人間が照合する待ちとして使い分ける。
- `park済み` は捕捉に退避済みで、捨てたという意味ではない。解決したら掃除 step で `[→DEC <ID>]` / `[done]` 化または削除する。

### 着地場所（sink）と着地条件（gate）

- **確約（旧 Sink 1）**：確定したことは、どのフェーズ・どのサーフェスで決まっても home に着地させる。横断＝`00-hub/DECISIONS_ja.md` に1行（ID・状態・なぜ／却下案つき）＋直すスポークのどの節をどう書き換えるか。局所＝該当スポークの節。
- **捕捉（park・旧 Sink 2）**：未確定・今やらない芽は、捨てずに捕捉（NOTES）へ park する。hub は `00-hub/NOTES_ja.md`、dev セッションは各リポの `NOTES_ja.md`（gitignore ローカル、`2026-06-20-03`）。dev NOTES に park した横断の芽は、セッション終いに hub へ **escalation-sweep** する（踏み忘れ対策は `dev-repo-protocol_ja.md` §NOTES に明文化）。
- **Gate (b)（着地妥当性）＝ 確約 ↔ 捕捉 の振り分け器**：確定は、それを**検証可能なサーフェスでのみ** 確約に有効着地できる。実装の現実に依存する判断（動かさないと決められない系）は、検証できないサーフェス（例：claude.ai）で「確定」として確約に落とさない。検証する（動かす）か、捕捉に park して検証可能なサーフェスへ送る。**サーフェス非依存の条件**＝特定ツールの特性でなく「検証可能性」で決まる（実行境界 `CLAUDE.md`／`2026-06-22-03` 同根）。
- **Gate (b) の観測単位と主張の規律（`2026-07-19-05`）**：「検証可能」は人・モデル名にも固定しない。**主体 × テーマ × 現在地 × 作業 × 利用可能な道具**ごとに観測し、権限境界・実行能力境界・検証能力境界を分ける（[llm-agent-boundary-guide](llm-agent-boundary-guide_ja.md) §8）。自己申告・自信は判定根拠にしない。実行者がいても誰も結果を検証できない確定は確約に落とさず、スコープ縮小・証拠追加・委譲・捕捉への park・工房での試行へ振り分ける。確約する主張は、検証したスコープ・証拠・未検証境界を明示し、**主張のスコープを証拠のスコープから越境させない**。
- **工房（tinker）＝判断しない領域（`2026-07-16-06`）**：独立公開リポ `mc-remote-tinker`。ID 無し・「なぜ」無し・Gate 無し・履歴保証無し。壊れたまま放置してよく、捨ててよい。sink ではない（何も着地せず、むしろ湧く）。唯一の義務はセッション終いの「発見あったか？」自問と hub NOTES への1行 sweep＝**一方向弁**（工房 →発見→ hub NOTES →Gate→ DECISIONS）。工房は正本ではなく、dev リポ・教材・設計文書から「使えるもの」として参照・依存されない。持ち込み禁止は knowledge と同一（秘密の実値・child safety）＋公開出力への OSS 線引き（`2026-06-15`）。
- **往復削減（助言）**：実装検証が必要とわかっている判断は、最初から対象を検証できる surface で始めると往復が減る。これは Gate と違い法ではない。

### ライフサイクル (c)

芽は `捕捉 →（決定）→ 確約 →（削除）` で動く。未決を解決したら捕捉から消す（NOTES 印を `[→DEC <ID>]`/`[done]` に、DECISIONS 未確定節は確定なら日付ログへ移動、スポーク §未決）。工房の芽はこのライフサイクルに乗らない（発見だけが sweep で捕捉に入り、そこから通常の経路に乗る）。手キュレーションの薄いビューなので、解決済みを消さないと優先度が陳腐化し、次の入口（「優先の未決は何だっけ？」＝`2026-06-19-05`/`2026-06-20-05`）が信用できなくなる。

### 入口：優先の未決から始める

「優先の未決は何だっけ？」で開く。読む先は未決ダッシュボード（`00-hub/NOTES_ja.md` のインボックス＝捕捉と、`00-hub/DECISIONS_ja.md`「未確定（起案・保留）」節＝理由が立った横断検討）と、各スポークの「未決事項」節（小さい未決）。ここから優先度でテーマを1つ選ぶ。

### 実行フェーズの戻り（非対称2系統）

実行は終端ではない。実行中に出た判断・芽は2系統で sink に戻り、どちらに落ちるかは Gate が振り分ける：

```text
（入口どこでも：ハブ/スポーク/実行サーフェス）
     │  渡り歩き・検討・試行（経路は自由・ループバック可）
     ▼
   判断/芽が出る ──Gate(b): 検証可能か？──┐
     │ Yes（確定）                        │ No / 未確定
     ▼                                    ▼
   確約                                  捕捉（park）
  home に着地                          NOTES に退避
  ・横断 → DECISIONS 1行                ・hub NOTES / dev NOTES
  ・局所 → スポーク §                  ・dev は終いに hub へ sweep
     │                                    ▲
     └──────── ライフサイクル(c) ─────────┤
        捕捉 →（決定）→ 確約 →（削除）    │
                                          │ 発見のみ1行 sweep（一方向弁）
  工房（tinker・判断しない・mc-remote-tinker）
```

- **protocol 波及に注意**。確定が通信契約に触れるもの（例：デバッグモードの結果返信＝protocol マイナー増分）は、`10-protocol`（SSOT）の更新まで含む。1スポークで閉じないことがある。

### 書き戻しと衛生（確約への着地手順）

```text
①確定（成果物を「確定搬送票」で出力。head/非リポ側は「中身だけ」＝ID 採番と commit/staging は渡さない＝2026-06-22-03）
   ↓
②repository-owning surface が ID 採番・配置 → 該当スポーク＋横断なら DECISIONS に1行
   ↓
③コミット／push
   ↓
④着地確認依頼を元リポへ戻し、元リポ側で着地を確認する
   ↓
⑤取り組んだ未決を捕捉から落とす（ライフサイクル c）
   ↓
⑥claude.ai のナレッジを差し替え（手動同期）
```

人間が運ぶ部分で迷い・要約揺れ・コピペ漏れが入りやすいので、dev repo 側からは正本の固定ブロックをそのまま渡す。
この票は「ナレッジへ入れる中身」を運ぶためのもので、DECISIONS の ID、採番、staging、commit 手順は書かない。

票の本文（フィールド一覧）は [dev-repo-protocol](dev-repo-protocol_ja.md) の `確定搬送票` ブロックを唯一の正本とし、
ここには再掲しない（DECISIONS `2026-07-21-06`）。4 dev リポの bootstrap が同ファイルの runtime marker を直接取得する
ため、本文を二箇所に置くと片方だけ更新された票が流通し、着地側の照合が崩れる。書式が要るときは正本を読む。
claude.ai サーフェスでは、同ファイルがプロジェクトナレッジに入っていることがこのリンクの解決条件になる（同期は手動）。

`根拠/検証` は test evidence class を明記する。`unit/deterministic` はテストコード＋PASS コマンド＋commit で足りる。
`live-auto` は protocol ratify / release gate の根拠に使う回だけ transcript を保存する。`live-human`
（`/mcremote pair` 等、人間操作入り）は `14-evidence/records/<test_id>_ja.md` の evidence record と sanitized
artifact を保存し、record path・artifact path・commit/SHA を書く。ローカル未追跡ファイル名だけの「証跡」は正式根拠
として受理しない（DECISIONS `2026-07-06-03`）。
開発フェーズ終端・milestone・公開準備では、再現コストだけでなく token cost と redaction / 移管コストも見て棚卸しする。

ナレッジリポ側の agent は、搬送票をレビュー・必要ならディスカッションした後、ID 採番・配置・ステージ・コミット・push
まで閉じる。完了後は、[dev-repo-protocol](dev-repo-protocol_ja.md) の `着地確認依頼` 固定ブロックを人間に返す。
人間はこれを元リポ側の agent に貼り、着地確認を依頼する。票の本文は同ファイルを唯一の正本とし、ここには再掲しない
（DECISIONS `2026-07-21-08`）。
`knowledge commit（push 済み SHA）` と `リモート反映確認` は必ず埋める。未記入は仮記入扱いで正式受理しない。

元リポ側の agent は、着地先を読み、元の実装・検証結果・搬送票と照合する。一致していれば `着地確認OK` を返し、
per-repo `NOTES_ja.md` の該当行を `[→DEC <ID>]` / `[done]` へ更新または削除する。ずれがあれば黙って直さず、
`着地差分メモ` として「何が違うか」「推奨修正」「影響」を返し、再度ナレッジリポへ持ち込める形にする。
これにより、ナレッジ着地後に元リポへ戻るループが明示され、空中に浮いた確定や park 済みメモの置き忘れを防ぐ。
人間入り実機テストを根拠にした着地確認では、元リポ側は evidence record / artifact が実在し、redaction 後の内容が搬送票の
主張（例: error 受信と close の順序）を支えているかを確認する。

### セッションクローズ（引き継ぎ）

セッションを閉じるとき、または長時間作業の節目では、[dev-repo-protocol](dev-repo-protocol_ja.md) の
`セッションクローズ票` 固定ブロックを短く出す。票の本文は同ファイルを唯一の正本とし、ここには再掲しない
（DECISIONS `2026-07-21-08`）。目的は、次回の自分・別 agent・別 LLM が作業状態を再開できること。
これは決定の正本ではないので、確定事項は `確定搬送票`、未確定の芽は NOTES、横断決定は DECISIONS に落とす。
セッションクローズ票は、それらへの入口と作業状態の要約に限る。

突発終了に備え、5時間制限・停電・ネットワーク不安・長い検証の前後では、完全な終了時でなくても途中版を出してよい。
ただし、票を保存用 SSOT にしない。次セッションへ残す必要がある未完了は per-repo `NOTES_ja.md` へ、横断するものは
hub `NOTES_ja.md` / `DECISIONS_ja.md` へ移す。`/compact` は、必要な NOTES 書き戻しまたはセッションクローズ票を出してから行う。
