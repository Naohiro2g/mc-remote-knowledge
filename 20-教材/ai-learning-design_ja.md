## 1. 教育哲学（前提として最重視するもの）

本プロジェクトの学習支援は、一貫した軸の上に立つ。AIアシスト方針はこの軸から逆算する。

```text
言語力 > 技術力
  読解力・聴解力・言語化力の育成が、技術力育成より優位にある。
  「言語化できること」が最重視される（最も難しいが、最も重要）。

構築主義
  文法を一から演習で積む方式の「正反対」。
  ある程度複雑なサンプルアプリをまず動かす → 改造する
  → その過程で文法を掴む。

判断履歴の重視
  コードの束や zip ではなく、設計記録1枚を渡す。
  「なぜそうしたか」を最重視する。
```

これらを通す学習の流れは、**観察 → 痕跡を残す → 言語化 → 比較・検証 → 自己判定 → 共有・再検証**。
観察は言語化の前提であり、まだ言葉にならない違いを、言語化できないという理由で捨てない。学習支援の目的は
生徒を教師が評価しやすくすることではなく、生徒自身が「何を観察し、何を根拠に、できた／まだできていないと
判断すればよいか」を獲得することに置く。

## 2. 中心原理: なぜAIアシストとこの哲学が噛み合うのか

AIアシストは上記3軸すべてと**構造的に一致する**。これは偶然ではなく、設計の土台。

```text
言語力 ↔ AI へのプロンプトも生成物の評価も、すべて言語行為
  → AI 時代、言語力は「技術を動かす力」そのものになった。
  → 言語力を先に据えていたことが、図らずも最適解だった。

構築主義 ↔ AI は「複雑なサンプルを即座に用意する」のが得意
  → 「まず動く複雑なものを与える」コストが激減。
  → 構築主義の最大の難点（良質教材の用意・学習者が複雑さに溺れるリスク）を
    AI が生成と伴走で解く。
  → AI が構築主義を初めて実用的にした。

判断履歴 ↔ AI が生成を肩代わりした後、人間に残るのは判断
  → 「なぜそうしたか」こそ人間の領分、という構造が一致。
```

## 3. 「細部把握の必要性」への回答（逆説）

「AI 時代に細部を全て把握する必要があるか」は、2つの異なる問いの混同。

| 問い | AI の肩代わり | 必要性 |
| ------ | -------------- | -------- |
| コードを生成・記述するための細部把握 | できる | 劇的に低下 |
| 生成物を判断・検証・統合するための細部把握 | できない | むしろ上昇 |

**消えるのは「暗記・記述の労力」、消えないのは「判断のための足場」。** AI を使いこなす人ほど土台の理解が深く、AI に使われる人は判断できず生成物を鵜呑みにする。後者の価値（判断の足場）は希少化して上がる。

## 4. 設計原則（各プラットフォーム共通）

```text
AIにコードを生成させてよい。
ただし学習者は『なぜそうなるか』を言語化できること。
= 生成は任せ、判断は学習者が握る。
```

**補足（誤読防止）**：これは「AI に生成させない」ではない。生成はむしろ委譲し歓迎する（§2＝AI が構築主義を初めて実用的にした、§5 Python 層＝AI の出番は最大）。握るのは判断・検証・統合・言語化。後述の「答えを手渡さず問い返す」（本節の壁打ち・[debug-session-design]）は**チューター対話の局所設計**であって、生成そのものの禁止ではない。両者を潰すと「生成は任せ、判断を握る」が「生成させない」に反転する。

これは「判断履歴重視」哲学の学習版。各層で「AI に渡してよい部分」と「学習者が握るべき部分」の境界が、学習目的によって異なる位置にある。**一本の方針で全層を貫こうとすると失敗する。**

**補足（2026-06-30）: 全ては投影で扱う。** 正本は密な SSOT に置き、学習者・貢献者・開発者が受け取るものは、
言語、作業対象、現在の熟達度、利用している LLM / サーフェスに合わせてその場で投影する。同じ人でも、ある対象では
プロ、別の対象では入門者になりうるため、適応単位は「人」ではなく「テーマ × 現在地 × 作業」。コーディングも
LLM アシスト前提なので、教材・開発・OSS 貢献・ナレッジ運用は分断せず、同じ投影原理でつながる。

### 補助原則: 「AIアシストに振りすぎない」

- エージェント型 AI は便利すぎて、学習者から主体性を奪いやすい（主体が AI に移る）。
- インライン補完型（従来の Copilot 的）は「学習者が書こうとしている途中の補助」で、主体は学習者に残る。構築主義・言語化と相性が良い。
- → 教育用途では、デフォルトで全部を AI に振らない設計を意図的に行う。どこまで任せるかを教材ごとに設計する。

### AI を「言語化の壁打ち相手」として使う

AI は答えを出す道具としてだけでなく、**言語化を引き出す個別チューター**として使える。

```text
育てる使い方 ↔ 奪う使い方
  「なぜ?と問い返す AI」 ↔ 「答えを即座に出す AI」
  同じツールが、設計次第で正反対の効果を持つ。
```

「生成は任せ、判断は握る」原則が、この分岐の防波堤になる。とくに低年齢層では言語化のハードルが高いため、「問い返す」対話設計が要となる。

### 観察から自己判定までを支える

「なんか動かない」は誤った報告ではなく、まだ分解されていない観察である。学習者へ最初から原因説明や長い文章を
要求せず、次の順で足場を置く。

```text
何が起きると思ったか
  ↓
実際に何が見えたか
  ↓
どこが違うと気づいたか
  ↓
その違いを残す（スクリーンショット・ログ・値・短いメモ）
  ↓
次に何を見れば切り分けられるか
  ↓
何を根拠に、できた／まだできていないと判断したか
```

スクリーンショット等は文章を書けないときの劣った代替ではなく、**解釈を要求する前に観測を保存する低い入口**。
「言語要求を下げる」ことと「思考要求を下げる」ことを混同しない。AI は原因や点数を先に返す評価者ではなく、
「何が見えたか」「何が違ったか」「次に何を見るか」を問い、観察の切り出しと言語化を助ける鏡として使う。

自己評価は感想や自己採点ではない。学習者が作品について主張し、根拠となる観察を選び、未確認部分を区別できる
ことを指す。教師・AI が正誤を返し続けるのでなく、正しさの判定権と検証方法を学習者へ返す。

### 創造性を支える制約

自由度は、それだけでは創造性を生まない。現在のスコープを把握し、外枠を見る／細部へ潜る／別のスコープへ移る／
再接続する、という可逆な操作を支える適度な制約を置く。教師が進行・評価を揃えるための管理的制約と、学習者が
注意を配分して探索・深化するための創造的制約を区別する。

## 5. Scratch と Python の並列ビークル

Scratch を Python への一方向の踏み石として扱わない。両者は、同じ protocol と同じ world に別の認知入口から触れる並列の vehicle である。

- Scratch は、見える object と並行動作から因果を観察しやすい。
- Python は、生成・抽象化・再利用・検証を言語で組み立てやすい。
- Scratch で prototype を作り、Python で発展させ、結果を同じ world で比較できる。
- 並行性を Scratch 固有の魔法にせず、identity が main / substream を持つ共通 model へ写す。

土台の contract は `1 stream = 1 connection = 1 build state`（`2026-06-24-01`）。player identity が main stream と複数 substream を持てる方向を `2026-07-21-07` で起案している。Scratch stage / sprite / clone をどの stream へ写すかは b4 scope freeze 前に確定する未決であり、確定済みとして教材へ焼かない。
