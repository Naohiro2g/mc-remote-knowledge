# 設計記録: Scratch 実行モデルの理解まとめ

> マイクラリモコンの Scratch クライアント設計（接続状態・ライフサイクル・並行ストリーム）が依拠する、Scratch の実行モデルの理解を一箇所に固める文書。b2 ライフサイクル（[versioning §10.11.1](../10-protocol/versioning-design_ja.md) 項9）・接続ブロック冪等（DECISIONS `2026-07-04-03` 却下⑥）の前提意味論を支える。
>
> 出典: claude.ai 設計セッション（Scratch 実行モデルの理解精密化）→ 確定搬送票 2026-07-04・repo 着地 2026-07-05。基礎（§1）は人間の設計説明からの構成、精密化（§2）は搬送票の verbatim。

## 1. 基礎（イベント駆動・並行・断片実行）

- Scratch は**イベント駆動・並行**のシステム。実行状態は様々なイベントで変わる。
- 実行状態を変えるトリガーの3分類:
  - (1) 緑の旗がクリックされたとき（or Enter キーが押されたとき。Enter＝緑の旗は Scratch 1.4 のプレゼンテーションモードの挙動で、Scratch 3 stock には無い＝§2 項6）
  - (2) ブロック、あるいは連結されたブロック（スタック）をクリックしたとき。**コードエリアにあるとき・パレットエリアにあるときの両方**。
  - (3) 赤い停止ボタンがクリックされたとき
- (1)(2) では非実行状態なら実行状態へ。(3) では実行状態なら停止状態へ。
- **Jupyter Notebook と似た点**＝コードの断片を部分的に試せる。セルごとに実行ボタンがある感覚。
- **緑の旗＝プログラム開始をあえて前面に出さない**。必要なときは緑の旗による実行で初期化（だけ）を行う、という考え方。狙いは、初期化が必要な場面が多く一見差が無くても、**イベント駆動型への意識を不当に下げないため**。他のトリガーによる初期化もできる。

## 2. 精密化（搬送票 verbatim・2026-07-04）

1. **クリックはトグル**。コードエリアで実行中のスタックをクリックすると停止する（非実行→実行の片方向遷移ではない）。パレットクリックは実行のみ＝コードエリアと非対称。
2. **旗押し直しは re-fire**。実行中に緑の旗を押すと旗ハット配下のスクリプトは停止→先頭から再起動。b2 ライフサイクル表（[versioning §10.11.1](../10-protocol/versioning-design_ja.md) item 9）の「旗押し直し」イベントと同一意味論であり、接続ブロック冪等（DECISIONS `2026-07-04-03` 却下⑥）はこの re-fire を前提とする。
3. **グローバル実行状態は存在しない**。実行状態は per-script（スレッド単位）の総和でしかない。起動は常に per-script、停止ボタン（stop all）だけが全域操作、という非対称。
4. **手動ジェスチャとイベント語彙の分離**。旗クリック／ブロッククリック／停止ボタンはエディタ上の手動ジェスチャの列挙であり、イベントシステムの本体はハットブロック群（キー押下・スプライトクリック・メッセージ受信・クローン開始・閾値ハット等）。緑の旗はハット群の対等な一員にすぎない。
5. **Jupyter アナロジーの差分2点**。(a) Scratch は断片実行ですら並行（別スタックを止めずに新スタックが走る）のに対し Jupyter のセルは直列。(b) Scratch には「カーネル再起動」に相当する明示操作が無い（リロードが最寄り）。
6. **「Enter＝緑の旗」＝検証済み（2026-07-05）**。**Scratch 1.4 の（少なくとも）プレゼンテーションモードで Enter キーが緑の旗と同じ扱い**という前例がある。一方**素の Scratch 3 にはグローバルショートカットは無い**認識（旗ボタンにフォーカスがある時の標準ボタン挙動としての Enter は別。TurboWarp 系フォークは追加可能）。自フォークではどちらにも定義できる位置ゆえ「現行 Scratch 3 の性質」としては書かず、必要なら 1.4 の前例を根拠に fork で復活させる選択肢として扱う。〔搬送票では要検証マーク付き、2026-07-05 に人間が 1.4 前例を確認して解消〕
