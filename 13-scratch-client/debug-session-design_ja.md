# 設計記録: デバッグセッション + AI組み込み（Python/Scratch クライアント）

> マイクラリモコン設計記録
> 関連: WS2（Scratch クライアント / wss ブリッジ）、WS3（param_mc_remote / 通信仕様）、WS1（AI 活用方針）
> 結節点: 本構想は3ワークストリームが1点で交わる結節点。[versioning-design] の通信プロトコル、[platform-design] のセッション基盤、[ai-learning-design] の AI 組み込み（方向B）の最初の実験場。

---

## 1. 出発点: 現状の学習ループの「片目」問題

```text
現状:
  Python でコードを書く
    → マイクラアプリで世界を覗いて結果を見る（視覚的フィードバック ○）
    → だが、エラー・内部状態は管理者しか見えないサーバーログに出る（学習者には不可視 ×）
```

学習者は「動いた／動かない」は目で見えるが、「**なぜ**動かないか」が見えない。言語力（とくにエラーの言語化）を最重視する本プロジェクトの哲学（[ai-learning-design]）からすると、これは致命的な欠落。エラーの言語化は言語力訓練の宝庫。

---

## 2. 既存の通信前提

Python だけで考えると、**結果返信はすでに mc_remote 通信に含まれている**:

- GET 系コマンド（座標→ブロック取得など）は結果を返す。
- つまり「返信チャネル」は新規ではなく、既存通信の拡張として設計できる。

問題は「常に詳細を返すと重くなる」点。素早い建築ができなくなる。→ **デバッグモードが必要。**

---

## 3. デバッグモードの設計

### 粒度（決定方針）

```text
- セッションごとのスイッチ（on/off）
- 各コマンドごとのレベル指定
```

2つの軸を併用。セッション単位で大枠を切り替え、コマンド単位で詳細度を制御。

### トレードオフの容認

- デバッグモードでは**建築速度の低下を容認する**（詳細返信のコスト）。
- 通常モード（高速建築）とデバッグモード（詳細フィードバック）を明確に分ける。常時返信はしない。

### 返す情報

- 結果コード（成功／失敗）。
- エラー（現状サーバーログにしか出ないもの）。
- Python 側のエラー処理も教材的に面白い素材になる。

---

## 4. 別セッションでブラウザに見せる構成

返信先を、**Scratch クライアントと同様のセッション構成**にする。

```text
Python 実行
   → mc_remote 通信（デバッグモード）
   → wss ブリッジ（既存 Scratch fork のインフラを転用）
   → セッション
   → ブラウザ画面で学習者が結果・エラーを見られる
```

既存の Scratch fork アーキテクチャ（`wss://bridge.mc-remote.com` → セッション）を、**Python 学習者向けデバッグビューに転用**する。インフラの素地が既にある。

> 整理が必要な点: Python のエラー処理を「Python 側で完結して教える」のと「ブラウザの別セッションで見せる」のは役割が異なる。両者の住み分けは未決（§8）。

---

## 5. 飛躍: セッションが双方向なら「実行環境」になる

「情報を見る」だけのつもりが、双方向セッションなら「建築コマンドも送れる」。

```text
ブラウザ上のセッション画面が:
  ├─ Python コードを書ける（簡単な建築）
  ├─ 実行結果・エラーが返ってくる（デバッグビュー）
  ├─ マイクラの世界と連動
  └─ ★ここに AI アシストを組み込める★
```

→ Python 入門の最初の一歩を、**VS Code 不要・ブラウザ完結**で踏める。環境構築の壁がないため、年齢層を下げられる。

---

## 6. なぜここで AI 組み込み（方向B）が生きるのか

[ai-learning-design] の「組み込み型 AI は範囲を制御できる層で生きる」が、ここで論理的に成立する。

```text
範囲が狭い（簡単な建築）       → AI の振る舞いを制御しやすい
セッション制御下              → エラー・状態を AI が把握できる
ブラウザ完結                  → 環境構築の壁がない（低年齢にも降ろせる）
デバッグ情報が AI に渡る       → 「なぜ動かないか」を AI と学習者が一緒に考えられる
```

決定的な違い: VS Code + Copilot では AI は「コードだけ」を見る。この構成なら、**AI が「コード + 実行結果 + マイクラの状態」を同時に見られる**。エラーログという「言語化の素材」を、AI と学習者の対話の中心に置ける。

→ [ai-learning-design] §4「AI を言語化の壁打ち相手として使う」「なぜ?と問い返す」設計の、理想的な最初の実験場。

---

## 7. 構想の位置づけ（3ワークストリームの結節点）

```text
WS2（Scratch クライアント / wss ブリッジ）
   → そのインフラを Python 学習デバッグビューに転用

WS3（param_mc_remote / 通信仕様）
   → デバッグモードの結果コード返信 = 通信プロトコルの拡張（[versioning-design] の protocol マイナー増分に該当）

WS1（グランドプラン / AI 活用方針）
   → このセッション画面が AI 組み込み（方向B）の最初の実験場
```

3つが1点で交わる。だからこそ重要度が高い。

## 8. 設計論点（着手時に詰める）

### 論点1: デバッグモードの粒度

- on/off の2値か、レベル制（エラーのみ / 全結果 / 詳細）か。
- 建築速度とのトレードオフをどう刻むか。
- （現方針: セッション単位スイッチ + コマンド単位レベル）

### 論点2: 結果コード返信の通信設計

- 既存の mc_remote 通信に「返信チャネル」をどう乗せるか（GET 系は既存）。
- Scratch の wss セッションと共通化できるか、別系統か。
- protocol バージョンとしてどう増分するか（[versioning-design]）。

### 論点3: ブラウザ実行環境の射程（最も悩ましい）

- 「簡単な建築」の範囲をどこまでにするか。
- VS Code 版 Python 学習への「橋」として、どこで線を引くか。
- **ブラウザが便利すぎると、構築主義の本丸（VS Code + 複雑サンプル）へ進む動機が消える問題。**
- 案: Scratch が「Python 入門の前段階」であるように、このブラウザ実行環境を「VS Code Python の前段階」として明確に位置づけ、機能をあえて絞る。

### 論点4: AI 組み込みの最小実験

- この画面で、まず何を試すか。
- 「エラーが出たら問い返す」の最小プロトタイプは何か。
- [ai-learning-design] の A→B 段階論のうち、ここが B の最初の着地点になりうる。

### 論点5: Python エラー処理教材との住み分け

- 「Python 側でエラー処理を完結して教える」教材と、「ブラウザ別セッションで結果を見せる」構成の役割分担。

## 10. 設計メモ（protocol b1 議論からの昇格・2026-06-27）

出典: claude.ai 設計議論 × ナレッジ突き合わせ 2026-06-26（DECISIONS `2026-06-27-04`・[wire-format-design](../10-protocol/wire-format-design_ja.md) §7③）。

### ライブ画面 = 全 b を貫く背骨

「ライブ画面」（＝本文書のデバッグセッションの学習者向け呼称）は、機能として特定 bN に割り当てるのでなく **b1 のログビューアから b5 まで育つ横串（観測点）**。Scratch/Python/Java など入口が増えても、接続状態・実行ログ・catalog・AI 補助・チュートリアルを**同じ場所**に集約する全クライアント共通ハブ。Scratch→Python 移行でも同じライブ画面を使い続けることで環境の連続性を作る。**ブランド名（固有名詞・言語非依存）＝WireScope（JP 表記: ワイヤースコープ）**（旧「Debug Session」硬名は引退）。**説明的通称（一般名詞・各言語）＝EN: Live session/screen ・JP: ライブ画面**＝学習者に意味が通る別称。学習者・貢献者は日本人に限定しないのでブランドは言語非依存・通称のみ各言語化＝落とし所表記 `WireScope (Live screen)` / `ワイヤースコープ（ライブ画面）`。実装は observer UI package `@mc-remote/live`（モノリポ `mc-remote/live`・**配置予約のみ／実装 bN**、DECISIONS `2026-06-27-08`）。

**setBlock の非同期エラーはここに落ちる**：b1 protocol は setBlock/setBlocks/chat.post を id 付き同期 request として扱い、疎通確認と error 観測を優先する（`2026-07-01-08`）。一方、サーバ→クライアントの **async error/log push（§7③ 推し=(c)）をライブ画面で受けて非同期表示**する形は別物＝push 用に Python の readline ループを改造する重い部分なので **bN（debug 統合）**。send-only / fire-and-forget の既定化やライブ画面のモードトグル（デバッグモード=setBlock に id 付与で同期応答 等のスイッチ）は bN/debug 側で設計する。ライブ画面側は b1 では持ち込まず、疎通は id 付き setBlock / getBlock で確認する。

### building 制限の wire 部分（`build_denied` 方針メモ）

多人数初学者環境（100×100 の箱庭が並ぶ）で暴発的な大規模破壊を物理的に不可能にする安全装置/教育的ガードレール。`permission_denied`（認証）とも `unknown_block`（入力ミス）とも層が違う認可エラー。`build_denied` は b1 の id 付き request で同期 error として返る world-state reason（[wire §7.3](../10-protocol/wire-format-design_ja.md)、DECISIONS `2026-07-01-08`）。

- **実行は all-or-nothing 維持**（setBlocks で1ブロックでもはみ出したら丸ごと実行しない）。partial（はみ出し分だけ拒否）にはしない＝多人数・初学者・隣接箱庭の3条件下では「中途半端な書き込みが隣の境界に残らない安全」が「壁が見える分かりやすさ」より重い。
- **分かりやすさは実行でなくエラーで実現**：実行は全部止め、error の `data` で「許可範囲 `bounds` と はみ出し座標 `violating`」を返し、ライブ画面ではみ出した面を可視化＝partial の教育的メリット（壁が見える）を world を汚さず回収。共同境界は all-or-nothing（安全）、個人箱庭内では partial 的手触りを将来探る余地。
- 教育原則「なぜそうなったかを言語化できること」（[ai-learning](../20-教材/ai-learning-design_ja.md) §1）に直結＝all-or-nothing＋範囲返却は理由を明示、partial は world を見て推測させる。エラー切り分け教育の素材。
